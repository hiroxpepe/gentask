## ✅ このセッションで到達したこと（データモデルの概念）
- **中心はタスクではない。** 世に出る成果物と物語が中心。タスクは末端の工程。
  → 世のタスク管理が全部ダメなのは「タスクを中心に置く」から。
- **content が最上位の包含者。** マンガ / アプリ（ゲーム）/ 書籍 / 3Dモデル販売＝リリース区分。
  task が結びつく先は content。
- **各 content は固有の概念群を持つ**（無理に共通化しない・固有のまま）。
  マンガ＝story/edition/episode、アプリ＝version/stage、書籍＝章/巻。
  → 共通の親（content）＋種別固有の子（クラステーブル継承）。
- **全 content 共通の骨格＝content / release / deliverable / asset。**
  根拠：全種別のリリース実態（LINEマンガ/GPlay/Kindle/BOOTH、JPEG/apk/EPUB/fbx）が表せる。
- **deliverable＝リリースの単位。** release が指す先。
  「リリースの単位になるか」が deliverable と asset を分ける線。3Dモデル販売では fbx が deliverable。
- **asset＝再利用単位のラベリング。** ファイル（画像/fbx/音声/テクスチャ）そのものではない。
  「水着エミリー」のような意味あるまとまりを括り、追跡するために要る。
  asset＝画像ではない（画像そのものなら概念不要）。素材の分解はキリがないので実務の粒度で止める。
- **deliverable ⇔ asset は多対多。中間テーブル**が「どの deliverable がどの asset を使うか」を持つ。
  含有（所有）ではない（含有なら再利用できない）。画像の再利用もこの中間テーブルで表す。
- **1 SQLite モノリスに集約。** マイクロサービス（Database per Service）不採用。
  販売管理 SaaS（freee 等）不採用。基幹システムへ育つ射程はあるが、実装は締切を回す最小から。
