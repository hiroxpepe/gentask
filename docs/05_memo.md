### 🚨 忖度抜きのアーキテクチャ・レビュー（致命的ツッコミ5連発）

#### 1. ID消失問題：`move_task` による「記憶喪失」の連鎖
Google Tasksには「リスト間のタスク移動API」が存在しないため、`insert` + `delete` で移動を再現する仕様になっています。これは**タスクIDが新しく生まれ変わる**ことを意味します。
これによって以下の致命的な連鎖崩壊が起きます。
* **Undo（スナップショット）の崩壊:** `snapshot.ts` のキーを `taskId` に直したとしても、週次スライドで「来週分」から「今週分」に昇格（移動）した瞬間、旧taskIdに紐づくUndo履歴は完全にロストします。
* **リンク切れリスク:** スライド時の `schedule_promoted_tasks` でカレンダー側との双方向リンクを再確立する設計にはなっていますが、もし同期エラーが起きたら最後、Google CalendarとTasksは永遠に結ばれない「孤立タスク」が爆誕します。

#### 2. 同期スコープの死角：「2週間」の壁で生まれるゾンビタスク
`sync` コマンドのカレンダー読取範囲が `timeMin: 2週間前` でハードコードされています。
もし、作者が極度のスランプや急病に陥り、特定のタスク（特に優先度の低いATASKなど）が**2週間以上放置された場合、そのタスクは同期の監視対象から外れ、永遠に自動更新されないゾンビと化します。** 「システム側が人間に合わせる」はずが、システムが人間を見捨ててしまいます。

#### 3. ハードコードされた「言葉遊び」の脆弱性
タスク配置やスライドの判定条件が、非常に脆い文字列一致に依存しています。
* スケジュール配置：タイトルが「プロット」を含むか、「ネーム」を含むか。
* 投稿完了チェック：CTASKのタイトルに「投稿」を含むか。

もしAI（Genkit）が気を利かせて「第42話 シナリオ構成」や「原稿アップロード」というタイトルでタスクを生成したらどうなるか？ カレンダーの配置ルール（水・木14時など）から漏れ、翌月曜の朝9時に容赦なく詰め込まれます。スライド処理も「投稿がない！」とパニックを起こして止まります。

#### 4. JSON埋め込みという「時限爆弾」
双方向リンクのために、タスクの `notes`（説明文）の末尾に `[gentask:{"eventId":"...","calendarId":"...","listId":"..."}]` を埋め込む仕様です。
スマホのタスクアプリ等から作者が**うっかり手動でメモを追記した際、このJSON文字列を1文字でも消してしまったり、改行でおかしくしてしまったら、同期システム全体がパニックを起こします。** 人間の自由な入力（エネルギーの赴くまま）を許容するシステム にしては、人間のヒューマンエラーに対してあまりに無防備です。

#### 5. 最大の矛盾：そもそも「CLI手動実行」は面倒くさくないのか？
「人間がシステムに合わせるのをやめる」と言いつつ、同期（`npm run sync`）や週次スライド（日曜21時の `npm run slide`）は**手動でコマンドを叩く前提**になっています。
極限状態で疲労困憊のクリエイターが、日曜の21時に「あ、スライドのコマンド叩かなきゃ…」と思い出すこと自体が、最大の精神的摩耗（タスク）ではありませんか？

### 💡 解決へのネクストアクション提案

ボロクソに言いましたが、ベースの設計思想が天才的なので、これらは実装の工夫で完全に乗り越えられます！

1. **メタデータ（拡張プロパティ）への依存強化:**
   タスクの `notes` ではなく、Google Tasksには隠しデータを持てないので、**「タスクID」ではなく「Gentask内での絶対不変のユニークID（UUID）」**を発行し、それをカレンダー側の `extendedProperties` と Tasks側の `notes` に埋め込む仕様に変更すべきです。これでTasksのIDが変わっても追跡可能です。
2. **Cron / GitHub Actions等での完全無人化:**
   `sync` は15分おき、`slide` は日曜21時に、バックグラウンドのサーバーレス環境（Cloud RunやGitHub Actions）で完全自動実行させるべきです。クリエイターはCLIすら触るべきではありません。
3. **AIへの「タグ付け」強制:**
   「プロット」や「投稿」といったタイトル文字の一致ではなく、AI（Genkit）がタスクを出力する際の `task_schema` に、厳密な `sub_role`（enum: `plot`, `name`, `post`, `other` など）を付与させ、システム側はその Enum のみで判定・配置を行うべきです。

さあ、この鬼レビューに対して、どこからコードを修正（あるいは反論）していきましょうか？ｗｗｗ 開発モード、いつでも全開でいけます！

まさにその通りです！完全に意図を理解しました。

「なぜわざわざMicrosoft 365を捨ててまで、GoogleカレンダーとGoogle Tasksにバックエンドを統一したのか？」という真の狙いが、システムを**Google Cloud（GCP）に着地させることで100%完全に回収**されます。

現在の「ローカルのNode.js/CLIで動かす」という構成は、あくまで動作検証のためのプロトタイプ（Phase 0〜6）ですよね。これをGCPにデプロイした瞬間、Gentaskは「手動の便利スクリプト」から「24時間365日稼働する真の自律型AIアシスタント」へと化けます。

具体的に、GCPで運用する「最強のメリット」は以下の4点に集約されます。

### ☁️ Google Cloud で実行する圧倒的メリット

* **1. Cloud Scheduler による「スライドの完全自動化」**
    日曜21時の `npm run slide` を手動で叩く運用 を、**Cloud Scheduler + Cloud Run Jobs** の組み合わせで完全自動化できます。これにより、クリエイターは「スライドのコマンドを叩き忘れる」という恐怖と精神的摩耗から永遠に解放されます。
* **2. 認証トークン地獄（OAuth）からの解放**
    ローカル実行では `.google_token.json` の生成やリフレッシュトークンの有効期限切れに悩まされますが、GCP上（Cloud Runなど）であれば、Google Workspaceの「ドメイン全体の委譲（Domain-wide Delegation）」を持たせた**サービスアカウント**を使用できます。面倒なOAuthのブラウザ認証フローを完全にスキップして、セキュアかつ永続的にカレンダーとタスクのAPIを叩き続けることができます。
* **3. Calendar Webhook による「リアルタイム同期（Sync）」**
    `sync` コマンドをわざわざ手動で叩いたり、定期実行でポーリングしたりする必要すらなくなります。Google Calendar APIの **Push Notifications（Webhook）** を Cloud Run で直接受け取るようにすれば、カレンダーの予定に「ok」や「undo」と書き込んだ**数秒後**には、裏でAIが走ってGoogle Tasksが更新されるという魔法のようなUXが実現します。
* **4. Vertex AI（Gemini）とのセキュアなネイティブ連携**
    システムの中核に Gemini 2.0 Flash（Vertex AI）を採用されていますが、これをGCP内で動かせば、APIキー（`GCP_VERTEX_AI_API_KEY`）を `.env` などで管理するリスクがなくなり、IAM（アクセス権限）だけでより安全・高速にAIを呼び出せます。

### 💸 GentaskをGCPで動かした場合のコスト内訳

* **Cloud Run（プログラムの実行環境）: ほぼ0円**
    毎月200万リクエスト、36万GB秒のコンピューティング時間が無料で使えます。作者が1日に何度かカレンダーを更新し、週に1回スライド処理を走らせる程度なら、無料枠の数パーセントも使い切りません。
* **Cloud Scheduler（日曜21時の定期実行）: 0円**
    1つの請求先アカウントにつき、毎月3ジョブまで無料で設定できます。「週次スライド」や「予備の定期同期」を設定する程度なら完全に0円です。
* **Google Calendar / Tasks API: 0円**
    企業向けの凄まじいリクエスト数（1日100万回など）を想定した制限になっているため、個人がAPIを叩く程度で課金されることはありません。
* **Vertex AI（Gemini 2.0 Flash）: 数十円〜百数十円 / 月**
    ここだけは完全無料ではなく従量課金になりますが、Gemini 2.0 FlashはAPI料金が破壊的に安いです。週1回のプロット生成（数千トークン）と、日々のカレンダー同期の意図解釈（数百トークン×数十回）を回しても、月の合計は数十円から、高くてもジュース1本分程度に収まるはずです。
