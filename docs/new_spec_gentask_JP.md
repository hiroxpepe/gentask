# Gentask 新仕様 — 会話駆動・スプレッドシート本体化

> 本ドキュメントは、Gentask の大改修方針を定めた仕様書である。
> 既存の Google Calendar / Google Tasks 連携を廃し、
> 「会話を進捗の源とし、スプレッドシートをタスク本体とする」構造へ作り替える。
> 現時点では骨格（設計方針）の確定にとどまり、列設計等の詳細は PoC を通じて詰める。

---

## 1. 改修の狙い

### 捨てるもの
現行 Gentask は、人間が Google Calendar に文字を書き込み（「ok」「手が止まった。明日やる」等）、
それを AI が読み取って Google Tasks を更新する構造だった。
これは「人間にカレンダーへ入力させている」点で、二重管理・入力口を残している。

### 軸・幹
本改修は、SC（FSC）タスク管理ダッシュボードの革新の本質をそのまま個人プロジェクトへ持ち込む。

> **入力させない・既存データを源にする。**
> 既存データを Single Source of Truth とし、ダッシュボードはそれを束ねて見せる／操作するビューに徹する。
> 人にどこかへ入力させる設計は、この軸に反するので採らない。

カレンダーへの手入力は「管理のための入力」だが、
LLM との対話は「作業の結果として自然に発生するデータ」である。
後者を源にすることで、二重管理を消す。

---

## 2. 新構造の骨格

```
  会話（進捗の源）
    │  ※Claude.ai で PoC。最終的に GitHub Copilot チャット。
    ▼
  LLM が会話を解釈
    │  会話文 → タスクの状態変化として解釈し、更新コマンドを生成・実行する
    ▼
  スプレッドシート（タスク本体 = Single Source of Truth）
    │  ※Google スプレッドシート
    ▼
  独自ダッシュボード（ビュー）
       スプシを読んで見せる／操作する。人に入力させる場ではない。
```

### 各要素の定義

| 要素 | 役割 |
|---|---|
| **会話** | 進捗シグナルの源。作業の流れで自然に発生する対話。管理のための入力ではない。 |
| **LLM** | 会話を解釈し、スプシのタスク行を更新する。会話 → 状態変化への変換を担う。 |
| **スプレッドシート** | タスク本体。Single Source of Truth。Google スプレッドシート。 |
| **ダッシュボード** | スプシを読んで見せる／操作するビュー。独自作成。入力口ではない。 |

---

## 3. 廃止するもの

- **Google Calendar**：入力口としても出力先としても完全に廃止する。
- **Google Tasks**：タスク本体をスプシへ移すため廃止する。
- **sync_flow（カレンダー自由記述の AI 解釈）**：会話駆動へ置き換わるため役目を終える。
- **notes 末尾への JSON メタデータ埋め込み（GENTASK_TAG 機構）**：
  スプシは「行 = タスク」で、UUID を独立した列に持てるため、notes への JSON 潜り込みは不要になる。

---

## 4. 活かすもの

現行 Gentask の「頭脳」は資産として残し、「手足」（Google API 連携）をスプシ＋ダッシュボードへ付け替える。

- **18sp / 36ブロックの制作モデル**：タスクの中身を定義する資産。維持する。
- **P/T/C/A の4象限（mode）**：タスク分類の軸。維持する。
- **task_schema（`lib/types.ts`）**：スプシ列設計の土台とする。
  現行フィールド：`title` / `mode` / `sub_role` / `priority` / `description` / `label` / `bucket`。
- **UUID 永続 ID**：会話でタスクを特定する鍵として維持する。行が動いても不変。
- **AI タスク生成（gen）ロジック**：出力先を Tasks/Calendar からスプシへ変える。
- **週次スライド（slide）ロジック**：スプシ上の行操作へ置き換える。

---

## 5. スプシ列設計の出発点

スプシの列は、既存 `task_schema` をそのまま写すことを出発点とする（詳細は PoC で確定）。
土台となる現行フィールドは以下。

| フィールド | 値 |
|---|---|
| `title` | タスク名 |
| `mode` | PTASK / TTASK / CTASK / ATASK |
| `sub_role` | plot / name / post / other |
| `priority` | 1〜9 |
| `description` | 背景・達成条件 |
| `label` | Red / Blue / Green / Yellow / Purple / Pink |
| `bucket` | current / next / done |
| `uuid` | 不変 UUID（v4） |

> 具体的な列の増減・型・期限（due）の持ち方などは、この段階では確定しない。
> PoC を動かしながら詰める。

---

## 6. PoC の進め方

- **場**：Claude.ai。
- **目的**：会話 → LLM 解釈 → スプシ更新の一連が成立することを実証する。
- **最終形**：源を GitHub Copilot チャットへ寄せる（作業場所と会話場所を VS Code 上で同一化する）。

> Claude.ai での PoC を経て Copilot へ寄せる、という移行順序自体はまだ確定していない。
