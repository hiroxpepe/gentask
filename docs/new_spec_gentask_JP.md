# Gentask 新仕様 — 人間がツールに仕えないタスク管理

> 本ドキュメントは、Gentask の大改修方針を定めた仕様書である。
> 既存の Google Calendar / Google Tasks 連携を廃し、
> 「時間の配置は LLM が提案しツールが確定し、人間は喋るだけ、事後に現実へ塗り替える」構造へ作り替える。
> データ本体は GitHub リポジトリ内に置く（正本＝SQLite の .db、窓＝CSV の両方を git 管理）。
> データの操作は、LLM を内蔵しない専用ツール（Gentask 本体）が決定的に担う。
> Gentask の目的は、**月間連載を続けられる状態を支えること**（→第2章・第7章・第8章）。
> 現時点では幹（設計思想）と骨格の確定にとどまり、スキーマ・実装の具体は次セッションで詰める。

---

## 0. Gentask とは何か — タスク管理の主従を元に戻す

タスク管理は本来、人間が忘れないように・段取りに悩まないように、
その負担を外部へ預けるための仕組みだったはずである。ツールが人間に仕える。それが本来あるべき姿だ。

ところが既存のタスク管理・タイムブロッキング製品は、いつのまにか主従を逆転させた。
人間がツールに合わせて入力し、配置を守り、ズレを手で直す——人間がツールに仕える側に回されている。
だから続かず、放置され、死ぬ。（実際、著者は M365 Loop/Planner/To Do/Outlook で同じ運用を試み、
毎日の手入力に疲れて 2025-08-07 に頓挫した。）

**Gentask は、この主従を元に戻す。**
人間は生きて、作業して、喋るだけでいい。
配置も、現実との辻褄合わせも、記録も、すべて Gentask の側（会話を解釈する LLM と、それを受けて動くツール）が引き受ける。
これは新奇な機能ではなく、タスク管理が本来そうあるべきだった姿への回帰である。

---

## 1. なぜ作り直すのか — 既存タスク管理が失敗する理由

既存製品は「人間が配置を守る」ことを前提にしている。配置するのも守るのも、ズレたら直すのも人間。
現実は配置どおり進まない（神回で倍かかる、詰まる、そもそもだるくてやらない）。
ズレるたびに手で直す苦痛から、やがて放置され、「現実と乖離した予定の墓場」になって死ぬ。

**根本原因は「未来を時刻に貼る」こと。** 未来のタスクをカレンダーの特定時刻に置くと、
その時刻に守れず、ズレ、手直しが要る。M365 の頓挫もこれだった。
**Gentask は「人間は配置を守らない」を織り込み、未来を時刻から解放する（→第4章）。**

---

## 2. 幹（設計思想）

**Gentask の目的：月間連載を続けられる状態を支えること。**
第0章の「主従を元に戻す」は Gentask が拠って立つ普遍の思想であり、本章の「月間連載を支える」はそれを実現する当面の具体目的である
（思想が幹、連載が最初の実（み）。まず著者一人の連載で思想が機能することを実証する）。
かつて週刊で 4コマ漫画を仕上げていた頃は充実していた。締切があり、毎週コンテンツが PNG として世に出て、
連載が続いていたからである。開発（3D・ゲーム）に移って前進はしたが、完成が遠く区切りがなく、充実は薄れた。
Gentask は、この「毎月コンテンツを完成させ世に出す連載のリズム」を、開発と両立する形で取り戻すための道具である。

> **時間の配置は LLM が提案しツールが確定する。人間は守らない。会話で現実を伝える。事後に現実へ塗り替える。**
> **塗り替えた記録は裁かず、後で苦笑いするために残す。世の中に無駄な時間は一つもない。**

- **配置する（未来・予定）**：制作モデル（18sp/36ブロック、P/T/C/A、エネルギー高低、工程の型）を
  理解して、LLM（会話）がざっくり配置を提案し、ツールが確定する。時刻には縛らない（→予定・カンバン形式、第4章）。
- **人間は守らない**：サボる、違うことをする、YouTube を見る。それを咎めない。
- **会話で現実を伝える**：人間はどこにも入力しない。ただ喋る（申告する）。それが現実を伝える唯一の源。
- **事後に塗り替える（過去・実績）**：LLM が会話を解釈して反映アクションに変換し、ツールが実績の枠を確定する（→実績・カレンダー形式、第4章）。
- **裁かない・苦笑いのためのログ**：記録は反省でも教材でもなく、
  「あー、こんなに YouTube 見てたかー」と後で苦笑いするためのもの。
  無駄な時間は一つもないという前提に立つ。裁かないからシステムは苦痛を生まず、死なない。

## 3. 時間の分類（決定的・人間が定義する）

時間はまず「作業」と「生活」に分かれる。**作業（個人事業）は別枠の本丸。** 生活はそれ以外。
分類は決定的でなければならない（LLM の判断に委ねない）。人間が定義しきる。

### 3-1. 作業（個人事業を積む）— 本丸
「しなくても死なないが、あえてやる」時間。成果物を生む。

| 記号 | 意味 | エネルギー |
|---|---|---|
| P | 企画（思考・戦略・言語化・計画） | 高 |
| T | 技術（技術検証・環境構築・実装・手順確立） | 中 |
| C | 制作（制作・デザイン・手作業・コンテンツ作成） | 低 |
| A | 管理（運用・管理・事務・ルーチン） | 随時 |

### 3-2. 生活（それ以外）
| 記号 | 意味 | 原理 |
|---|---|---|
| D | 生活（食事・入浴・睡眠・日常） | しないと死ぬ |
| J | 仕事（本業＝生きるための金を得る手段） | しないと死ぬ |
| W | 運動（散歩・懸垂。あえて体を動かす） | しなくても死なないが、あえて |
| M | 移動（通勤・電車等。着くための手段） | 拘束されるが頭は空く＝活用対象（耳読書・思考） |

> D/J/M は M365 実運用（Outlook 予定表）で塗れた実績がある。W は本仕様で新設。
> 「徒歩でイオン」「通勤中に Claude」のように 1 つの時間に複数の性質が混じる場合がある。
> **その割り切れなさを人間に背負わせない。** 人間は生の事実を申告し、盤面への反映は Gentask が行う（第5章）。
> ただし分類ルールそのものは決定的に定義され、LLM が気まぐれに分類を決めることはない。

### 3-3. 抽象（work / life）と実態（8値）の関係
「作業」「生活」は抽象、実態は 8 つの記号（P/T/C/A/D/J/W/M）。
型としては、作業カテゴリ（P/T/C/A）と生活カテゴリ（D/J/W/M）を独立に定義し、
その和が全体の 8 値となる。上位区分（work/life）は 8 値から導出する（データとして二重に持たない）。
移動（M）は「削減対象」ではなく、LLM 時代においては移動中に LLM へ指示を出すデフォルト時間＝新しい生産時間と捉える。

## 4. 二つの器 — 予定（カンバン形式）と実績（カレンダー形式）

時間軸は 1 本。だが未来と過去で粒度も器も異なる。**未来はざっくり、過去は精密。**
器は役割で呼ぶ（予定／実績）。表示形態としてカンバン／カレンダーの形をとるが、これらは Google 製品ではなく形式の呼称。

### 4-1. 予定（カンバン形式・未来・ざっくり・時刻なし）
未来のタスクはカードとしてカンバン形式で積む（「今週」「来週」「いつか」等の列）。
**時刻には貼らない。** ゆえに守れなくても崩れない。既存ツールが未来を時刻に貼って死んだのを避ける。
LLM がざっくり配置を提案し、ツールが確定する対象。

### 4-2. 実績（カレンダー形式・過去・精密・時刻あり・15分の枠）
実際に手を動かして申告した結果が、15 分の枠（slot、→第5章）としてカレンダー形式で落ちる。
**最小の枠＝15分。** 1 時間＝4 枠。SP は 15 分の整数倍（sp1=4, sp2=8, sp3=12, sp5=20, sp8=32）。
過去は確定しているので時刻に貼っても嘘にならない。各枠は P/T/C/A または D/J/W/M で分類される。

### 4-3. 流れ
予定にカードが積まれる（未来） → 人間が「3D やった」と申告する →
そのカードが実績の枠として落ち、**このとき初めて時刻が確定する**（過去）。
やる前は時刻なし（予定）、やった後に時刻あり（実績）。

**苦笑いの正体＝予定と実績のズレ。** 予定に残ったままのカード（やらなかったこと）、
実績の想定外の枠（YouTube 等）。この落差を後で眺めて苦笑いする。
（※これは「予定シート／実績シート」の 2 層ではない。1 本の時間軸を、未来は予定・過去は実績として見せるだけ。）

## 5. データモデルと器

### 5-1. 四つの実体（オブジェクト）
| 名前 | 概念 | 役割 | 主なキー・関係 |
|---|---|---|---|
| **content** | 作品 | 4コマ1話・3Dモデル・ゲーム等。連載の単位、PNG になって世に出る成果物。充実の源。 | uuid |
| **task** | 作業（工程） | content を作るための工程（プロット/ネーム/3D 等）。 | uuid / content_uuid（属す先） |
| **slot** | 枠 | 確保された 15 分の時間区画（放送業界の「枠」。放送前も後も枠）。分類（8値）を持つ。 | uuid / category（8値） |
| **assignment** | 割り当て | どの task が・どの slot で・実際どうだったか。task と slot の多対多を結ぶ中間。**LLM が会話で塗り替える対象。** | uuid / task_uuid / slot_uuid |

関係は多対多：1 content が複数 task を持ち、1 task が複数 slot で実行され、1 slot に複数 task が混じりうる。
その多対多を assignment が結ぶ。各実体は不変の uuid を持ち、参照キーとする。
（各プロパティの厳密な定義は第14章の宿題＝zod スキーマ確定で詰める。）

### 5-2. 器 — SQLite（.db）＋ CSV、両方を git に置く
3 層 × 多対多は素朴な 1 枚 CSV の限界を超える。ゆえに器はこうする：

- **正本＝ SQLite（.db、バイナリ）**：整合性・多対多・JOIN を担う。git に置き、リセットからそのまま復元する。
- **窓＝ CSV（テキスト）**：.db から吐き出す。git diff が効き、**git 履歴が苦笑いログを兼ねる**。ダッシュボードも CSV を読む。
- 番人ツール（第6章の操作層）が **commit 時に .db と CSV を必ずセットで更新**し、両者の整合を保つ。
- データ量は生涯でも数十 MB 規模で、容量は問題にならない。

### 5-3. 実装方針
- 言語は **TypeScript**。型は **素の zod**（旧 gentask の genkit は使わない）。
- **抽象（work/life）と実態（8値）** は、作業 enum（P/T/C/A）と生活 enum（D/J/W/M）を独立に定義し、
  その和で全体を作る。上位区分は値から導出（第3-3章）。
- **gentask 本体は LLM を内蔵しない。** LLM 連携（API 呼び出し）を持たない。CSV/SQLite を操作する CLI ツールに徹する
  （sfdc-user-tools / zcrm-tools と同じ「LLM は外、ツールは決定的」構造）。
- **LLM の推論を信用しない。** ゆえにロジックは単体テストで担保する（TDD。旧 lib/types.test.ts の流儀を継ぐ）。

## 6. 役割分担 — 判断（会話）と実務（ツール）を分ける

比喩すればゲーム：人間はプレイヤー（行動を宣言）、LLM は宣言を解釈する進行役、ツールが盤面を動かす。
ルール（分類 P/T/C/A・D/J/W/M、15 分の枠）は決定的に固定され、LLM はそれを勝手に変えない。

| 層 | 担当 | 役割 |
|---|---|---|
| 判断層 | LLM（会話＝Claude/Copilot。Gentask に内蔵しない） | 会話（申告）を解釈し、決定的ルールに従って反映アクション（提案）を返す。分類ルール自体は変えない。データを直接触らない。 |
| 操作層 | Gentask 本体（ツール） | LLM のアクションを検証し、正本（.db）と窓（CSV）を安全に・セットで更新。整合性を守り、git に commit。 |
| データ層 | .db（正本）＋ CSV（窓） | リポジトリ内。git 履歴（CSV の diff）が苦笑いログを兼ねる。 |

**分類の決定も実務も LLM に委ねない。** LLM がやるのは「会話を、決まったルールの反映アクションに変換する」ことだけ。
その変換結果すら鵜呑みにせず、操作層が検証してから盤面へ落とす（→第14章の関門）。

## 7. 編集者エージェントの駆動原理 — なぜ動くのか

かつて週刊連載を続けられた漫画家は、才能ゆえではない。**編集者がいたから**である。
手塚治虫ですら、編集者なしには描き続けられない。作家一人では、才能があっても連載は続かない。
伴走する他者がいるから続く。**Gentask が目指すのは、この編集者である。**

ただし、編集者を丸ごと再現するのは魔法であり、できない。LLM は反応装置で、
作家を追いかける・缶詰にする・締切前に電話する、といった能動的な伴走はできない（→第8章の限界）。

そして、より本質的な点。**編集者が原稿を取りに来たのは善意ではなく、編集者自身に都合（利害）があったから**だ。
雑誌に穴を空けられない、自分の仕事がかかっている。作家が落とせば編集者が困る。だから必死になる。
編集者の能動性の正体は、編集者自身の利害だった。素の LLM はマスターがサボっても困らない。
**LLM に欠けているのは、能力以前に「動機（都合）」である。**

そこで、編集者エージェントに擬似的な都合（効用）を持たせる。

- **効用**：「連載を落とさせたくない」を効用関数として持つ。
- **緊張度**：締切までの余裕で効用の緊張度が変わり、それが働きかけの温度になる。
  余裕があれば「そろそろですね」、逼迫すれば「先生、落ちます！」。
- **裁かず泣きつく**：「あなたが悪い（裁く）」ではなく「私が困る（泣きつく）」。ゆえに圧にならず、可笑しみで人を動かす。
- **キャラクター**：無味乾燥な通知ではなく、都合を持ったキャラとして振る舞う（STUDIO MeowToon の感性と地続き）。

この「都合が行動を駆動する」機構は、ゲームのエネミー AI（敵にも都合がある）と同じ構造であり、
Utility AI（Animo に通じる思想）から来ている。
**ただし Animo は思想の源泉であって、現時点では別プロジェクト・別実装。** Gentask が Animo 本体に依存する意味ではない。

> 本章は今セッションの到達点であり、駆動原理の骨子。効用関数の具体・緊張度の閾値・キャラの実装は次セッションで詰める。

## 8. 行動のトリガー — 何が人を動かすか

LLM（編集者）は反応装置で、能動的に追いかけられない（第7章）。だから、人間が自ら動き出すトリガーが要る。
モチベーションを外から煽るのは採らない（達成できないと罪悪感の圧になり、裁かない思想に反する）。
代わりに、次の三つで動機を生む。

- **体重計モデル（日次）**：実績のカレンダー（15分の枠）を毎朝見る。体重計が「痩せろ」と言わずに数字を出すだけで
  行動を促すように、埋まった／スカスカの実績を見せるだけで、本人が自ら動く。裁かず、促さず、しかし行動が起きる。
  ——**実績を見ること自体が最大のトリガー**であり、これがダッシュボードの中心目的（第12章）。
- **締切スプリント（週次）**：週一の区切り（締切）を置く。締切効果は M365 で実証済み（週刊投稿に変えたら回った）。
  ただし達成をコミットさせない（守らせない・裁かない）。区切りが来たら slide が実績を締め、カンバンを繰り上げるリズム。
  月間連載（大きな締切）を、週次スプリントで刻んで締切効果を効かせる。
- **編集者の泣きつき（随時）**：効用が逼迫した編集者エージェントが「先生、落ちます！」と泣きつく（第7章）。

## 9. 動作フロー — 起点から記録まで

LLM は反応装置で、自ら時刻を刻めず、リポジトリを常時監視もできない。この限界に合わせ、**起点は「会話が始まったこと」自体**とする。
瞬間的な反応は求めない。1 回の会話で、次の流れが一巡する。

1. **起点（会話の開始）**：会話の入り口で、リポジトリを一度読む（clone/pull）。正本（.db）を手元に得る。
   外部から降ってくるタスクはリポジトリ（Issue や inbox 等）に投げ込まれており、ここでまとめて拾う。startup.sh 運用と噛み合う。
2. **申告**：人間が喋る（「3D やった」「plot 詰まった」等）。これが現実を伝える唯一の源。
3. **変換**：LLM（会話）が申告を、決定的ルール（分類・枠）に従った反映アクション（提案）に変換する（第6章 判断層）。
4. **検証**：ツール（操作層）が、変換結果を鵜呑みにせず検証する（参照先の実在、値が 8値の範囲か等）。
   通らなければ人間に確認を返す（→第14章の関門）。LLM の推論を信用しないための関所。
5. **確定・記録**：検証を通ったものだけ、ツールが正本（.db）と窓（CSV）にセットで書き、git に commit する。
   実績の枠が確定し、git 履歴に苦笑いログが積まれる。
6. **区切り（週次）**：締切スプリントの区切りで slide が実績を締め、予定（カンバン形式）を繰り上げる（第8章）。

この一巡はすべて「人間は喋るだけ、判断は LLM の変換のみ、実務と整合性はツール」で回る（第6章の役割分担）。

## 10. 廃止するもの

- Google Calendar / Google Tasks（本体・入出力として全廃）。※予定・実績の器は「カンバン／カレンダー形式」と呼ぶが、これは形式の呼称で Google 製品ではない（第4章）。
- sync_flow（カレンダー自由記述の AI 解釈）。
- notes 末尾への JSON メタデータ埋め込み（GENTASK_TAG 機構）。
- 予定／実績を分ける 2 層（ベースライン）。時間軸は 1 本。
- Google スプレッドシート案（本体はリポジトリ内に置くため）。
- genkit（Google の LLM フレームワーク）＋ Gemini。gentask 本体は LLM を内蔵しないため不要（第5-3章）。型は素の zod。
- LLM 内蔵アプリとしての旧 gentask 構造。「LLM は外、ツールは決定的」へ作り替える。

> 補足：SQLite は当初「バイナリで git diff が取れず苦笑いログを失う」として却下していたが、
> 「.db（正本）＋ CSV（窓）を両方 git に置き、番人ツールが同時更新する」構成（第5-2章）で採用に転じた。

## 11. 活かすもの

- 18sp / 36ブロックの制作モデル、P/T/C/A ＋エネルギー高低（配置の知能の中核）。
- sub_role の工程の型（plot/name/post）→ task の工程分類に。
- UUID 永続 ID（content/task/slot の参照キー／会話でのレコード特定）。
- gen（AI 配置）ロジック → カンバンへの配置提案に。
- slide（週次スライド）ロジック → 週次スプリントの締めと繰り上げに（→第7章）。
- sync_action_schema の設計思想（LLM は変換アクションを返す、検証と実操作はツール）。

## 12. ダッシュボード

予定（カンバン形式）と実績（カレンダー形式）を読んで見せる／操作するビューに徹する。人に入力させる場ではない。
**中心目的は、実績（15分の枠）を毎朝見せて行動のトリガーにすること（体重計モデル、第8章）。**
苦笑いのためのログ（実績の git 履歴）を眺める窓でもある。構造はゲーム的だが、UI をゲーム化する意味ではない。

---

## 13. 既存製品との差別化

「LLM に話してタスク/予定を登録する」ものは OSS・製品ともに複数存在する（Notion+LLM、AnythingLLM 等）。
だが、それらは「未来の予定登録」が中心。Gentask の固有性は次の 4 点の組み合わせにある：

1. 未来の予定でなく **過去の実績を会話で刻む**（＋未来はカンバンにざっくり）。
2. **15 分の枠で 1 日全体を塗る**（網羅性）。
3. **生活込みの意味分類**（P/T/C/A × D/J/W/M。散歩・通勤・懸垂も入る）。
4. **リポジトリが核・git 履歴が苦笑いログ**。

この 4 点を薄めると「よくある LLM タスク登録ツール」に転落する。ここが存在理由である。

## 14. 未解決の宿題（次セッションの起点）

**今セッションで確定したこと**：命名（content/task/slot、枠=slot）、器（.db 正本＋CSV 窓、両方 git、番人が同時更新）、
抽象/実態の型（work/life と 8値、B 案）、LLM 非依存の CLI ツール、テストで担保（TDD）。

**残る宿題**：
- content / task / slot / 割り当ての zod スキーマ確定（プロパティの詰め）。※次の着手点。
- 予定（カンバン）と実績（カレンダー）の区別を、どのフィールドで持つか（slot か assignment の状態/時刻の有無で表す）。
- 申告を「決定的に」盤面へ落とす関門の具体（LLM の変換結果をツールがどう検証し、怪しければ人間に返すか）。
- 編集者エージェントの効用関数・緊張度の閾値・キャラの具体（第7章の続き。Animo は思想源泉・別実装）。
- 「その他（O）」の恒久箱を作るか（原則反対。作るなら「未分類の一時保留」に限定）。
- 申告のコスト（喋り忘れは仕様。将来の音声入力は構想の隅に留保）。
