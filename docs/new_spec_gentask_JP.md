# Gentask 新仕様 — 人間がツールに仕えないタスク管理

> 本ドキュメントは、Gentask の大改修方針を定めた仕様書である。
> 既存の Google Calendar / Google Tasks 連携を廃し、
> 「LLM が時間を配置し、人間は喋るだけ、LLM が事後に現実へ塗り替える」構造へ作り替える。
> タスク本体は GitHub リポジトリ内の CSV に置き、CSV の安全な操作は専用ツール（Gentask 本体）が担う。
> 現時点では幹（設計思想）と骨格の確定にとどまり、CSV の具体的な列設計は PoC を通じて詰める。

---

## 0. Gentask とは何か — タスク管理の主従を元に戻す

タスク管理は本来、人間が忘れないように・段取りに悩まないように、
その負担を外部へ預けるための仕組みだったはずである。ツールが人間に仕える。それが本来あるべき姿だ。

ところが既存のタスク管理・タイムブロッキング製品は、いつのまにか主従を逆転させた。
人間がツールに合わせて入力し、配置を守り、ズレを手で直す——人間がツールに仕える側に回されている。
だから続かず、放置され、死ぬ。

**Gentask は、この主従を元に戻す。**
人間は生きて、作業して、喋るだけでいい。
配置も、現実との辻褄合わせも、記録も、すべて Gentask（LLM =マネージャー）が引き受ける。
これは新奇な機能ではなく、タスク管理が本来そうあるべきだった姿への回帰である。

---

## 1. なぜ作り直すのか — 既存タスク管理が失敗する理由

既存製品（Todoist / Motion / Reclaim / Sunsama / Google カレンダー等）は、
いずれも「人間が配置を守る」ことを前提にしている。配置するのも人間、守るのも人間、
ズレたら手で直すのも人間。現実は配置どおり進まない（神回で倍かかる、詰まる、そもそもだるくてやらない）。
ズレるたびに手で直す苦痛から、やがて放置され、「現実と乖離した予定の墓場」になって死ぬ。

**Gentask は「人間は配置を守らない」を最初から織り込む。**

---

## 2. 幹（設計思想）

> **LLM が時間を配置する。人間は守らない。会話で現実を伝える。LLM が事後に現実へ塗り替える。**
> **塗り替えた記録は裁きに使わず、後で苦笑いするために残す。**

- **LLM が配置する（予定）**：制作モデル（18sp/36ブロック、P/T/C/A、エネルギー高低、工程の型）を
  理解した配置。機械的スケジューラの「締切と空きからパズルを埋める」とは異なる、意味を伴う配置。
- **人間は守らない**：サボる、違うことをする、YouTube を見る。それを咎めない。
- **会話で現実を伝える**：人間はどこにも入力しない。ただ喋る。それが現実を伝える唯一の源。
- **LLM が事後に塗り替える（実績）**：会話を解釈し、現実で予定を上書きして時間軸を1本にする。
  ——ゆえに「予定シート／実績シート」の2層は持たない。
- **裁かない・苦笑いのためのログ**：記録は反省でも教材でもなく、
  「あー、こんなに YouTube 見てたかー」と後で苦笑いするためのもの。
  裁き（圧）に使わないから、システムは苦痛を生まず、死なない。ゆえに塗り替えの痕跡は消さず積む。

---

## 3. 起点 — 会話の開始が起点、監視は不要

LLM は反応装置である。自ら時刻を刻めず、リポジトリを常時監視することもできない（機構上の限界）。
Gentask はこの限界を消そうとせず、限界に合わせて運用を設計する。

**起点は「会話が始まったこと」自体。瞬間的な反応は求めない。会話の途中で十分。**

外部から降ってくるタスク（上司のチャット、依頼、締切等）は、何らかの経路で
リポジトリ（Issue や inbox 等）に投げ込まれる。LLM はそれを監視しない。
**次に人間が会話を始めたとき、その入り口で、LLM がリポジトリを一度読む**（clone / pull で手元に来る）。
溜まったタスクをそこで拾い、時間の経過（前回からどれだけ過ぎたか）も読み、会話の中で捌く。

これは startup.sh 運用と噛み合う：新チャット開始 → clone/pull → その時点で全データが手元にある → 読む。
特別な監視機構は要らない。

---

## 4. データ構造 — 多対多（現実に忠実）

現実では、タスクと時間はきっちり対応しない。
1つのタスクは複数の時間ブロックにまたがり（3D モデリングは 6 ブロック）、
1つの時間ブロックで複数のタスクが混ざる（3D を進めつつ割り込みでメール）。
タスクと時間は独立して存在し、事後にゆるく結びつけられるだけ。**関係は多対多。**

予定はきれいな 1 対 1 に見えても、現実はぐちゃぐちゃな多対多。
その乖離こそが「あーこんなに YouTube 見てたか」の苦笑いの正体である。

### CSV 3枚（リポジトリ内）

| ファイル | 単位 | 役割 |
|---|---|---|
| `tasks.csv` | 1行=1タスク | タスクマスタ。何をやるか。`uuid`/`title`/`mode`/`sub_role`/`priority`/`description`/`label`。ここだけが「タスクの正典」。 |
| `blocks.csv` | 1行=1ブロック | 時間ブロック。いつ。時刻等。 |
| `assignments.csv` | 1行=1割り当て | 中間テーブル。「どのタスクを・どのブロックで・実際どうだったか」。多対多を結ぶ。**LLM が会話で塗り替える対象。** |

- `uuid`（不変）が tasks と assignments を結ぶ外部キーを兼ねる。
- 構造を持ちたいカラム（塗り替え状態等）には **フラットな JSON** を文字列で入れる（ネストしない）。
- 表示のときだけダッシュボードが加工する。**データはフラットに素直に、加工は見せる側の責任。**
- **git 履歴が塗り替えログ（苦笑いログ）を兼ねる。** ゆえに器はテキスト（CSV）であって、
  バイナリの SQLite は採らない（diff が取れず、読みの素直さも失うため）。

---

## 5. 3層アーキテクチャ — 限界に合わせた役割分担

LLM は賢いが雑（整合性を保証できない、生 CSV を正確に 1 行だけ書き換えるのが苦手）。
ツールは愚直だが正確。両者の限界に合わせて役割を分ける。

| 層 | 担当 | 役割 |
|---|---|---|
| **判断層** | LLM | 会話を解釈し「何をしたいか」を**アクション（構造化した意図）**として返す。生 CSV は触らない。反応装置に徹する。 |
| **操作層** | Gentask 本体（CSV ツール） | LLM のアクションを受け、多対多の CSV を安全に読み書きする。整合性を守り（存在しない uuid を弾く、片方だけ直す事故を防ぐ、1行だけクリーンに書いて diff を汚さない）、git に commit する。 |
| **データ層** | CSV 3枚 | tasks / blocks / assignments。フラット JSON セル。リポジトリ内。git 履歴が苦笑いログ。 |

**LLM に整合性を期待しない。ツールに判断を期待しない。** これが「反応装置の限界」への最終的な答え。
この「LLM はアクションを返し、実操作はコードが担う」設計は、
現行 Gentask の `sync_action_schema`（complete / reschedule / add_note 等を LLM が返し、
実際の更新はコードが行う）の思想の延長である。

---

## 6. 廃止するもの

- **Google Calendar / Google Tasks**：入力口・出力先・本体としてすべて廃止。
- **sync_flow（カレンダー自由記述の AI 解釈）**：会話駆動へ置き換わるため役目を終える。
- **notes 末尾への JSON メタデータ埋め込み（GENTASK_TAG 機構）**：CSV は行=レコードで uuid を独立列に持てるため不要。
- **予定／実績を分ける2層（ベースライン）**：現実で 1 本の時間軸を上書きするため採らない。
- **Google スプレッドシート案**：本体はリポジトリ内 CSV とするため採らない（読みの素直さと git 履歴のため）。
- **SQLite 案**：バイナリで git diff が取れず、苦笑いログと読みの素直さを失うため採らない。

---

## 7. 活かすもの

- **18sp / 36ブロックの制作モデル**：LLM が配置の意味判断に使う。
- **P/T/C/A の4象限（mode）＋エネルギー高低**：配置の知能の中核。
- **sub_role の工程の型（plot/name/post）**：配置ルールの土台。
- **UUID 永続 ID**：tasks と assignments を結ぶ外部キー／会話でのレコード特定の鍵。
- **gen（AI タスク生成）ロジック**：出力先を CSV へ変える。
- **slide（週次スライド）ロジック**：CSV 上の行操作へ置き換える。
- **sync_action_schema の設計思想**：LLM はアクションを返す、実操作はコード（第5章）。

---

## 8. ダッシュボード

独自ダッシュボードは、CSV 3枚を読んで見せる／操作するビューに徹する。人に入力させる場ではない。
フラットなデータを、表示のときだけ加工して見せる（時系列に並べる、色を付ける等はビューの責任）。
苦笑いのためのログ（塗り替えの git 履歴）を眺める窓でもある。

---

## 9. PoC の進め方

- **場**：Claude.ai。最終的に GitHub Copilot チャットへ寄せる。
- **目的**：会話 → LLM がアクションを返す → ツールが多対多 CSV を安全に塗り替える、の一連の実証。
- **最初に潰すべき点**：LLM が「アクション」を正しく構造化できるか、
  ツールが多対多 CSV（フラット JSON セル）を整合性を保って読み書きできるか。

---

## 10. この仕様の位置づけ

実装コードは、明確な「GO」指示があるまで着手しない。
