# Gentask 新仕様 — 人間がツールに仕えないタスク管理

> 本ドキュメントは、Gentask の大改修方針を定めた仕様書である。
> 既存の Google Calendar / Google Tasks 連携を廃し、
> スプレッドシートを本体とした「LLM が時間を配置し、会話で現実を伝え、
> LLM が事後に現実へ塗り替える」構造へ作り替える。
> 現時点では幹（設計思想）と骨格の確定にとどまり、
> 列設計等の詳細は PoC を通じて詰める。

---

## 0. Gentask とは何か — タスク管理の主従を元に戻す

タスク管理は本来、人間が忘れないように・段取りに悩まないように、
その負担を外部へ預けるための仕組みだったはずである。
ツールが人間に仕える。それが本来あるべき姿だ。

ところが既存のタスク管理・タイムブロッキング製品は、いつのまにか主従を逆転させた。
人間がツールに合わせて入力し、配置を守り、ズレを手で直す——
人間がツールに仕える側に回されている。だから続かず、放置され、死ぬ。

**Gentask は、この主従を元に戻す。**
人間は生きて、作業して、喋るだけでいい。
配置も、現実との辻褄合わせも、記録も、すべて LLM（マネージャー）が引き受ける。
これは新奇な機能ではなく、タスク管理が本来そうあるべきだった姿への回帰である。

---

## 1. なぜ作り直すのか — 既存タスク管理が失敗する理由

一般的なタスク管理・タイムブロッキング製品（Todoist / Motion / Reclaim / Sunsama /
Google カレンダー等）は、いずれも「人間が配置を守る」ことを前提にしている。

- タスクを時間ブロックに割り当てる（配置）のは人間。
- 配置どおりに動くのも人間。
- 現実がズレたら、人間が手でドラッグして直す。

現実は配置どおりに進まない。神回で倍かかる、詰まる、そもそもやらない（だるい）。
ズレるたびに人間が手で直す必要が生じ、その手入力が苦痛だから、やがて放置され、
「現実と乖離した予定の墓場」になって死ぬ。

**Gentask はこの前提を捨てる。「人間は配置を守らない」を最初から織り込む。**

---

## 2. 幹（設計思想）

> **LLM が時間を配置する。人間は守らない。会話で現実を伝える。LLM が事後に現実へ塗り替える。**
> **塗り替えた記録は裁きに使わず、後で苦笑いするために残す。**

### 2-1. LLM が配置する（予定）
時間ブロックへのタスク配置は、人間ではなく LLM が行う。
LLM は Gentask に組み込まれた制作モデル（18sp / 36ブロック、P/T/C/A の4象限、
エネルギーの高低、plot=水木14:00 / name=金14:00 等の工程の型）を理解した上で、
「この作業はエネルギーが要るから集中できる時間帯へ」といった意味を伴う配置をする。
既存製品の「締切と空き時間から機械的にパズルを埋める自動スケジュール」とは異なり、
制作の文脈を理解した配置である。

### 2-2. 人間は守らない
配置は「守らせるための命令」ではない。人間はサボる、違うことをする、YouTube を見る。
それが現実であり、Gentask はそれを咎めない。

### 2-3. 会話で現実を伝える
人間はどこにも入力しない。ただ喋る。
「今日は結局ずっと 3D いじってた」「plot 詰まって進まなかった」——
作業の流れで自然に発生する会話が、現実を伝える唯一の源。
PoC は Claude.ai で行い、最終的に GitHub Copilot チャットへ寄せる。

### 2-4. LLM が事後に塗り替える（実績）
LLM は会話を解釈し、時間シートを現実に合わせて書き換える。
「plot の予定だったブロックを 3D 作業で埋め直す」といった塗り替えを、人間の代わりに行う。
予定と実績を2つ並べて差分を睨むのではなく、現実で予定を上書きして1本にする。
——ゆえに「予定シート／実績シート」の2層は持たない。時間軸は1本。

### 2-5. 裁かない・苦笑いのためのログ
塗り替えた記録は、反省材料でも学習の教材でもない。
後で振り返って「あー、こんなに YouTube 見てたかー」と苦笑いするためのもの。
記録を裁き（「次は守れ」という圧）に使わないからこそ、システムは苦痛を生まず、死なない。
そのため、塗り替えの痕跡は消さずに積む（最新状態だけの上書き消去はしない）。

## 3. 既存製品との差別化

| 観点 | 既存製品 | Gentask |
|---|---|---|
| 時間ブロックへの配置 | 人間が手でドラッグ | LLM が制作モデルを理解して配置 |
| 配置を守る前提 | 守る前提（守れず破綻） | 守らない前提（織り込み済み） |
| 現実とのズレの修正 | 人間が手で直す | LLM が会話を聞いて事後に塗り替え |
| 記録の入力 | 人間が手入力 | 会話から自動で刻む（入力させない） |
| 記録の用途 | 反省・変動分析（裁き） | 苦笑いのためのログ（裁かない） |
| 予定と実績 | 2層で差分を比較 | 1本の時間軸を現実で上書き |

構造（時間シート＋タスクシートの2シート）自体は既存のタイムブロッキングにもある。
**革新は、盤面を動かす主体が「人間」でも「機械的スケジューラ」でもなく、
制作モデルを理解した LLM であり、かつ人間の不履行を前提に事後で辻褄を合わせる点にある。**

---

## 4. スプレッドシート構成（2シート）

タスク本体は Google スプレッドシートに置く。Single Source of Truth。
シートは2枚。役割で分ける（予定／実績では分けない）。

### 4-1. 時間シート（盤面兼ログ）— 幹
時間ブロックが単位。LLM が配置した結果が載り、会話に応じて LLM が現実へ塗り替えていく盤面。
塗り替えの痕跡を消さずに積み、後で苦笑いできるログを兼ねる。
「時間ブロックを記録すること」が Gentask の幹である。

### 4-2. タスクシート（作業の在庫）
作業（タスク）が単位。LLM が時間シートへ「何を」置くかの材料となる在庫。
1つの作業が複数の時間ブロックにまたがったり、複数回配置されたりするため、
時間シートとは別に持つ。
既存 `task_schema`（`lib/types.ts`）を列設計の土台とする：
`title` / `mode`(PTASK/TTASK/CTASK/ATASK) / `sub_role`(plot/name/post/other) /
`priority` / `description` / `label` / `uuid`（不変）。

> 各シートの具体的な列（時間シートがブロック開始時刻・割当作業・塗り替え履歴をどう持つか、
> タスクシートと時間シートをどのキーで結ぶか等）は、この段階では確定しない。PoC で詰める。

## 5. 廃止するもの

- **Google Calendar**：入力口としても出力先としても完全に廃止。
- **Google Tasks**：タスク本体をスプシへ移すため廃止。
- **sync_flow（カレンダー自由記述の AI 解釈）**：会話駆動へ置き換わるため役目を終える。
- **notes 末尾への JSON メタデータ埋め込み（GENTASK_TAG 機構）**：
  スプシは「行 = レコード」で UUID を独立列に持てるため、notes への JSON 潜り込みは不要。
- **予定／実績を分ける2層（ベースライン）という発想**：
  現実で1本の時間軸を上書きするため採らない。

---

## 6. 活かすもの

現行 Gentask の「頭脳」は資産として残し、「手足」（Google API 連携）を
スプシ＋ダッシュボードへ付け替える。

- **18sp / 36ブロックの制作モデル**：LLM が配置の意味判断に使う。維持。
- **P/T/C/A の4象限（mode）＋エネルギーの高低**：配置の知能の中核。維持。
- **sub_role の工程の型（plot/name/post）**：配置ルールの土台。維持。
- **UUID 永続 ID**：会話でレコードを特定する鍵。維持。
- **gen（AI タスク生成）ロジック**：出力先を時間シート／タスクシートへ変える。
- **slide（週次スライド）ロジック**：スプシ上の行操作へ置き換える。

---

## 7. ダッシュボード

独自ダッシュボードは、スプシ（時間シート・タスクシート）を読んで見せる／操作するビューに徹する。
人に入力させる場ではない。苦笑いのためのログ（時間シート）を眺める窓でもある。

---

## 8. PoC の進め方

- **場**：Claude.ai。
- **目的**：会話 → LLM 解釈 → 時間シートの配置・塗り替え、の一連が成立することを実証する。
- **最終形**：源を GitHub Copilot チャットへ寄せる（作業場所と会話場所を VS Code 上で同一化）。

> Claude.ai での PoC を経て Copilot へ寄せる移行順序自体は、まだ確定していない。
